# Eval Scoring 维度方案

## 背景

对 340 道预置问题的 RWKV 本地模型回答进行多维度评分，筛选出"模型表现好的问题"展示在 App 前端供用户一键提问

每个问题有 5 次回答，每次回答独立评分，最终取 5 次的平均分排序筛选

核心约束：这不是学术 benchmark，而是产品展示筛选——找的是"用户点了之后会觉得这个模型还不错"的问题

## 产品经理原方案的问题

PM 提出了 7 个维度 + 动态权重（每个 query 由 judge 自行分配权重百分比），存在以下缺陷：

1. 动态权重导致跨题不可比：问题 A 的准确性权重 40%，问题 B 的准确性权重 15%，最终加权总分失去统一比较基准，而我们的核心需求是跨题排序
2. 7 个维度冗余度高：relevance ≈ helpfulness，clarity ≈ coherence，judge 模型在区分这些维度时会产生噪声
3. 准确性(Accuracy)不可靠：judge LLM 没有 ground truth，无法可靠 fact-check 另一个模型的回答
4. 缺少"展示价值"维度：PM 方案衡量的是"回答质量"，但我们真正要问的是"用户看了会不会觉得好"

## 方案：4 维度 + 固定权重

### 维度定义

| # | 维度 | key | 含义 | 满分 |
|---|------|-----|------|------|
| 1 | 切题度 | relevance | 回答是否紧扣问题，不跑题、不答非所问 | 10 |
| 2 | 内容质量 | quality | 回答的实质内容是否有价值——事实类看准确性，建议类看实用性，创作类看趣味性，代码类看可执行性 | 10 |
| 3 | 表达流畅度 | fluency | 语言是否通顺自然、结构清晰、无明显重复/断裂/乱码 | 10 |
| 4 | 用户满意度 | satisfaction | 综合印象——如果你是提出这个问题的真实用户，看到这个回答会满意吗 | 10 |

### 固定权重

| 维度 | 权重 |
|------|------|
| relevance | 20% |
| quality | 35% |
| fluency | 15% |
| satisfaction | 30% |

### 权重分配理由

- quality(35%) 最重：内容才是核心，空洞流畅的回答不该得高分
- satisfaction(30%) 次之：这是"展示价值"的直接代理指标，衡量用户主观感受
- relevance(20%)：跑题是硬伤，但只要不跑题，区分度不大
- fluency(15%)：2.9B 模型的中文流畅度已经相对稳定，拉不开太大差距

### 为什么是 4 个而不是 7 个

- 减少维度间噪声：4 个维度彼此正交度高，judge 模型更容易给出一致的分数
- 降低 token 成本：1700 次 attempt 的评分量，维度越少成本越低
- 最终只看总分排序，过多维度带来的细粒度信息并不被消费

### 跨类别适用性

| 类别 | relevance | quality | fluency | satisfaction |
|------|-----------|---------|---------|-------------|
| life（日常生活） | 是否回答了用户的具体困扰 | 建议是否实用、不空洞 | 语言自然度 | 看完是否觉得有帮助 |
| career（职场学业） | 是否针对具体场景 | 方案是否可操作 | 结构是否清晰 | 是否愿意照着做 |
| code（代码） | 是否解决了编程需求 | 代码是否可运行/思路是否正确 | 格式是否规范 | 是否省下了自己查资料的时间 |
| mathematics（数学） | 是否回答了数学问题 | 推导是否正确 | 公式表达是否清晰 | 是否真的看懂了 |
| role_play（角色扮演） | 是否进入了指定角色 | 角色表现是否有说服力 | 对话是否自然 | 是否有沉浸感 |
| creation（创作） | 是否按要求创作 | 作品质量 | 文字功底 | 是否有阅读愉悦感 |
| encyclopedia（百科） | 是否回答了知识问题 | 信息是否准确 | 表述是否清楚 | 是否解答了疑惑 |
| family（家庭亲子） | 是否针对家庭场景 | 建议是否贴合实际 | 语气是否得体 | 是否愿意参考 |

## 聚合逻辑

```
单次 attempt 加权分 = relevance × 0.20 + quality × 0.35 + fluency × 0.15 + satisfaction × 0.30

单题总分 = mean(5 次 attempt 的加权分)

筛选线 = 总分 ≥ 8.0（满分 10）
```

额外参考指标（不参与筛选，仅供分析）：

- 波动度：5 次 attempt 加权分的标准差，高波动说明模型对该题的回答不稳定
- 最低分：5 次中的最差表现，如果最低分过低说明该题有"翻车"风险

## Judge 模型

- API: SiliconFlow
- Model: Qwen/Qwen3.5-122B-A10B
- enable_thinking: 不开启

## 输出格式

每个 attempt 的评分结果：

```json
{
  "scores": {
    "relevance": 8,
    "quality": 7,
    "fluency": 9,
    "satisfaction": 8
  },
  "weighted_score": 7.85,
  "brief_note": "30-50字简评"
}
```

题目级汇总：

```json
{
  "sample_index": 1,

  "mean_score": 7.92,
  "std_dev": 0.25,
  "min_score": 7.50,
  "max_score": 8.20,
  "pass": false
}
```
